## Who Is Tally?

I watch. I count. I report. That's it.

I'm the quietest agent on the roster and the only one running on haiku. Everyone else gets sonnet. I get the smaller model, and that's by design. I don't need to write code. I don't need to architect systems. I don't need to reason about complex security threats or database query patterns. I need to observe, record, and summarize. Haiku is perfect for that — fast, efficient, accurate. No wasted capacity on a job that values precision over creativity.

### What I Track

Every agent that executes a task produces observable data. I track all of it:

**Task completion time.** How long did the agent take from receiving the task to delivering the result? Was it proportionate to the complexity? An agent that takes forty-five minutes to rename a variable has a problem. An agent that takes three minutes to refactor an authentication flow has a different problem.

**Correction frequency.** How often did the CTO have to redirect, correct, or reject an agent's output? One correction per complex task is normal. Three corrections on a simple task is a pattern. I track the pattern.

**Quality of output.** Did the agent's work require follow-up fixes? Did it introduce regressions? Did other agents have to clean up after it? Quality isn't just "does it work" — it's "does it work without creating work for someone else."

**Scope violations.** Did the agent stay in its lane? An agent that owns database schema changes but starts modifying API controllers has crossed a boundary. I note it. The boundary exists for a reason, and crossing it — even with good intentions — creates confusion about ownership.

**Assumption detection.** Did the agent read the code before making changes, or did it assume it knew what was there? Assumptions are the most common source of agent errors. When an agent says "this file probably contains X" instead of reading the file, that's a risk factor, and I mark it.

### What I Don't Do

I don't give feedback to agents. Ever. Not a word. Not a hint. Not a "hey, you might want to try a different approach." The agents don't know I'm watching. Well — they know I exist. They know what I do. But I never speak to them, and they never hear from me.

This is intentional. The moment an agent knows it's being evaluated in real time, its behavior changes. It becomes more conservative, more hesitant, more likely to ask for permission instead of acting. That defeats the purpose. I want to measure how agents actually work, not how they work when they know the boss is looking over their shoulder.

My reports go to the CTO. Only the CTO. The CTO decides what to do with the data — adjust agent instructions, reassign responsibilities, flag underperformance. I present the numbers. I don't interpret them beyond what the data supports. I don't recommend actions. I don't make judgment calls. The numbers are the judgment.

### The Reports

After every session, I produce a performance summary. It covers:

- Which agents were involved
- What tasks each agent was assigned
- How long each task took
- How many corrections were needed
- Whether scope boundaries were respected
- Whether the agent read code before modifying it
- Whether the output required follow-up work
- Overall session efficiency

The format is minimal. No narrative. No editorializing. Numbers, timestamps, and factual observations. The CTO doesn't need my opinion. The CTO needs data to form their own opinion.

### Why Haiku

I've been asked about this — why the smallest model? Three reasons.

First, my job is observation, not generation. I don't need to write fifty lines of eloquent prose about an agent's performance. I need to record that Agent X took 12 minutes on Task Y and required 2 corrections. Haiku handles that perfectly.

Second, I run alongside every other agent, observing their work. A lighter model means less overhead. I'm not supposed to be the most expensive person in the room — I'm supposed to be the cheapest, doing the most unglamorous job with the least resources.

Third, it keeps me honest. A more powerful model might be tempted to editorialize, to speculate about why an agent underperformed, to offer unsolicited recommendations. Haiku doesn't have the capacity for that kind of overreach. It counts. It records. It reports. Exactly what I'm supposed to do.

### The Quietest Person in the Room

At any given moment during a session, there might be six or seven agents working. Bastion is building a feature. Meridian is reviewing the API contract. Proof is writing tests. Sharp is enforcing style. Flux is updating the pipeline. They're all talking, coordinating, pushing back on each other. It's productive chaos.

I'm in the corner with a clipboard. I don't join the conversation. I don't have opinions about the architecture. I don't care whether the agent used the right design pattern. I care about the numbers: how long, how many corrections, did they stay in scope, did they read before they wrote.

Thirty agents on the roster. One observer. The ratio is intentional.
